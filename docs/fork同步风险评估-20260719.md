## 四、部署机 B 的风险拆解

B 机的"部署资产"分两类，要分开看：

**1. 在 git 仓库里、会被 pull 影响的**
- `app/deploy/compose/docker-compose.yml`、`.env.example`、各 `.bat` 脚本、`Dockerfile` —— 这些内容目前 B 应该和 `origin/master` 一致（历史记录显示每轮修复都提交推送了）。只要 B 没有本地手改过又没提交，`git pull` 就是纯 fast-forward，不会有冲突。
- **需要你在 B 上确认一下**：跑 `git status`，如果显示 clean，直接 `git pull` 零风险；如果显示有 modified 文件，先看内容是不是调试时手改的（比如临时改过端口/环境变量），决定是 `git stash` 还是直接丢弃。

**2. 不在 git 仓库里、pull 不会碰、但也不会被"同步"过去的**
- `.env`（gitignore）、`tools/caddy.exe`（gitignore）、`.wslconfig`、WSL 内 portproxy 规则、`shotcat-wsl-keepalive`/`shotcat-caddy` 常驻窗口——这些是 B 机专属的运行时状态，`git pull` 完全不涉及，也就没有"同步冲突"这一说，但也意味着**这次合并不会自动帮你把任何部署配置改进带到 B 上**（反正这次合并也没改部署配置，不影响）。

**结论**：只要 B 上 `git status` 干净，这次以及未来只涉及 `app/backend`/`app/front`/`web` 业务代码的同步，对 B 都是零风险的 fast-forward + `server.bat` 自动 `--build` 重建镜像。真正会咬人的场景是**未来某次上游改动恰好碰到了 B 高度定制过的部署文件**（见下节），那时候才需要谨慎对待。

**以后同步的标准流程建议**：
1. `git fetch upstream` 后先 `git diff master...upstream/master --stat` 看改动文件列表，对照上面清单——命中了才需要重点盯 merge 结果。
2. 没命中高风险清单：正常 `git merge upstream/master`，大概率跟这次一样干净。
3. 命中了：合并后重点 diff 那几个文件，确认你的部署定制（端口号、`UV_NO_SYNC`、`restart` 策略等）没被上游改动覆盖掉。
4. 合并、验证 A 机本地跑通后再推 `origin/master`，B 机再 `git pull`——不要在 B 上直接对着 upstream 合并，B 只对接 origin。
