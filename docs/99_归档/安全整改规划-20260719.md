### 2026-07-19 阶段一完成情况

- **1.1**（2026-07-20 完成；原"直接删 ports"方案发现会破坏本地原生开发——backend 不进容器时要靠 localhost 连容器里的 mysql/redis/rustfs——经确认后调整如下）：
  - **base `docker-compose.yml` 不再发布 mysql/redis/rustfs 任何宿主机端口**（server.bat 部署场景零暴露）；本地开发所需的端口发布挪进新增的 `docker-compose.dev.yml` override，仅 install.bat/test.bat/run.bat 加载，server.bat 不加载。**迁移正式服务器时只带 base 文件，绝不把 dev override 带上生产。**
  - override 里实测**无法**用 `127.0.0.1:` 绑定——WSL2 NAT 的 localhost relay 只转发 0.0.0.0 监听（2026-07-20 实测，loopback 绑定时 Windows 侧 localhost 探测不通），故保持默认绑定；NAT 模式下 WSL 子网对 LAN 不可路由，实际仍只是本机可达。
  - **纵深防御：Redis 加 `requirepass`**（Redis 同时是 Celery broker，能连上就能调度/篡改任务）。`REDIS_PASSWORD` 必填、缺失时 compose 直接拒绝启动，healthcheck 改为带认证探测；`app/deploy/compose/.env` 与 `app/backend/.env` 已同步真实随机密码，install.bat 首次生成 backend/.env 时自动带上。
  - 验证：base 与 base+dev 两种组合 `docker compose config` 通过；Windows 侧 3306/6379/9000/9001 连通稳定（连续 3 轮探测全通）；Redis 无密码连接被拒、带密码 PONG；用 backend settings 实测 mysql/redis/rustfs 三条连接链全通；299 个测试全绿。
  - 排查副产物（既有环境行为，与本次改动无关）：无活动 WSL 会话时发行版会终止（`vmIdleTimeout=-1` 只保 VM 不保发行版），dockerd 和容器随之起停——验证或服役期间需保持一个 keepalive 会话（server.bat 已内置 `wsl -- sleep infinity`）。

## 阶段二：API Key 暴露修复（本次体检重点，明确优先于其余功能性风险）

### 2.1 现状确认（2026-07-19 复核结论）

- `Provider.api_key` / `api_secret`：`app/backend/app/models/llm.py:78-79`，**明文存储**，`String(4096)`，无加密。
- `ProviderRead` 响应 schema（`app/backend/app/schemas/llm.py:51-56`）**不包含** api_key/api_secret 字段——GET 接口不会把明文 key 回显给前端，这点是好的，不用重做。
- **真正的洞在这里**：`app/backend/app/api/v1/__init__.py` 里 `/llm` 路由只挂了"必须登录"（`dependencies=[Depends(get_current_user)]`），**没有所有者校验**。`app/backend/app/services/llm/manage.py` 的 `get_provider`/`update_provider`/`delete_provider`/`list_providers_paginated` 全部不区分 `created_by`。也就是说：**任何登录用户都能修改/删除任何其他用户创建的 Provider**，包括把 `base_url` 改成攻击者自己的服务器地址——下一次任何用户用这个 Provider 生成文本/图片/视频时，后端会把存量的**真实明文 api_key** 当 `Authorization: Bearer <key>` 请求头发到攻击者的服务器上（`app/backend/app/dependencies.py:56-92` `_ImageHttpRunnable` 的逻辑），相当于**用改 base_url 的方式把别人的密钥钓出来**。这是一条比"直接读密钥"更隐蔽、但同样严重的攻击链。
- `created_by` 字段目前由请求体客户端自己填（`ProviderCreate.created_by`，`app/backend/app/schemas/llm.py:24`），不可信，和 `Project.owner_id` 此前的问题一模一样。
- `cryptography>=42.0.0` 已经是现有依赖（`pyproject.toml`），但**全仓库没有任何地方真正用它加密过东西**——有依赖没实现。

- **3.3**：新增 `app/core/ssrf_guard.py`。攻击链已核实为间接可控：两个调用点的 url 都来自"LLM 供应商返回的生成结果"，而登录用户可把自己的 Provider 指向自己的服务器，返回内网 URL 即可让后端当跳板。防护三段式：①请求前 `getaddrinfo` 解析全部 IP，非公网一律拒（`not ip.is_global`，覆盖私网/回环/169.254 云元数据/CGNAT/IPv6 mapped）；②禁用重定向自动跟随、3xx 直接报错（顺带修掉旧行为把重定向空 body 落库的问题）；③响应到手后复核实际连接的对端 IP（`network_stream` 扩展），封堵 DNS rebinding 窗口。开关 `SSRF_ALLOW_PRIVATE_TARGETS`（默认 false）仅供本地 mock 供应商调试。**残余风险**：rebinding 场景下请求本身已发出（可能触发内网 GET 副作用），但响应内容不会回流；彻底根治需 pin-IP 连接，成本高，当前场景不做。

- **3.5**：复核结论——`main.py` 两个 handler 本身干净（未处理异常固定回 "Internal server error"，不含 str(exc)；FastAPI 未开 debug，Starlette 不渲染堆栈页）。全仓扫 `detail=...{exc}` 模式共 4 处：3 处是自家校验函数的受控文案（projects style 组合、shot_character_links upsert），不算泄露；**1 处真问题已修**——`services/studio/image_task_references.py` 曾把 boto3/存储异常原文拼进 400 detail（可泄露对象存储 endpoint/bucket），改为原文只进服务端日志、对外通用文案。实测：对真实 `app.main:app` 挂临时路由抛含敏感路径的 RuntimeError，响应体仅 `{"code":500,"message":"Internal server error",...}`。

#### 密钥轮换完整流程（迁移当天照做，呼应 M4）

1. **备份**：先对整库（至少 providers 表）做快照，rekey 脚本不可回滚。
2. **生成新值**：`AUTH_JWT_SECRET`＝`python -c "import secrets; print(secrets.token_hex(32))"`；`PROVIDER_SECRET_ENC_KEY`＝`python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`；`MYSQL_*`/`REDIS_PASSWORD`/`RUSTFS_ACCESS_KEY`/`RUSTFS_SECRET_KEY`/`CADDY_BASIC_AUTH_*` 各用随机值。
3. **先重加密再换钥（顺序不能反）**：用**旧** `PROVIDER_SECRET_ENC_KEY` 跑 `uv run python -m app.cli.rekey_provider_secrets --old-key <旧> --new-key <新>`，成功后再把 `.env` 的 `PROVIDER_SECRET_ENC_KEY` 改成新值。
4. **其余密钥直接换**：`AUTH_JWT_SECRET` 换值即可（存量 JWT 失效、用户重新登录，可接受，无需迁移数据）；`MYSQL_*`/`REDIS_PASSWORD`/`RUSTFS_*` 改 compose/.env 与 backend/.env 后重建容器；`CADDY_BASIC_AUTH_*` 改后重启 server.bat。
5. **重启并验证**：重启 backend/celery/caddy；验证登录、LLM 生成（能解出 api_key 正常调用）、文件上传下载。

- **4.4（账号体系升级，2026-07-20 产品决策：维持现状不做）**：评估过管理员角色 / 自助注册 / 找回密码三个方向，结论是当前"命令行建号 + JWT + 项目级 owner 隔离"对内部工具 + D+ 每公司独立实例的场景已够用：自助注册会削弱 Caddy 门禁（进了 basic_auth 就能自建号，需邀请码才安全）、找回密码依赖当前没有的 SMTP 邮件服务、管理员角色在账号数很少时收益不足以覆盖后端 role+接口+前端界面的成本。**留到有明确的自助管人/多角色管理需求时再重启评估**（届时优先做"管理员角色 + 管理员重置密码"，不做开放自助注册）。
