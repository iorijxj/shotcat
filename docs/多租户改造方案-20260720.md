# 多租户硬隔离改造方案

> 状态:方案定稿(2026-07-20),待按阶段执行
> 关联总规划:[公众化开发规划-20260720.md](公众化开发规划-20260720.md)(本方案是其中的骨干工作流 M2)

## 1. 背景与目标

shotcat 从"局域网内部工具"升级为**面向公众、部署到外部服务器、同时服务国内外用户**的产品。面向公众意味着陌生用户/多家公司会在同一套系统里共存,当前"扁平单用户 + 散落归属校验"的模型无法提供数据隔离。

**目标**:引入独立的租户(tenant)维度,把数据隔离从"散在代码里的三套 `assert_*_owned`"收敛成**统一、难绕过、默认安全**的租户隔离层,并为将来接入外部认证预留干净接缝。

## 2. 现状(改造前事实基础)

- **鉴权唯一出口**:`app/backend/app/dependencies.py` 的 `get_current_user`(读 Bearer → 解 JWT → 查 `User`)。全仓无第二处解 token,所有路由靠 `Depends(get_current_user)`。JWT payload 只有 `sub`(=`User.id`)/`iat`/`exp`。
- **路由门禁**:`app/backend/app/api/v1/__init__.py` 聚合层整组挂 `dependencies=[Depends(get_current_user)]`(film/llm/studio/script_processing 整组强制登录;仅 health、auth 公开)。
- **用户表**:`app/backend/app/models/auth.py` 的 `User`(`String(64)` 主键,只有 username+password_hash)。**无租户/组织概念**。
- **无正式迁移框架**:schema 靠 `app/backend/app/core/db.py` 的 `Base.metadata.create_all`;历史"迁移"是幂等 CLI(如 `app/cli/backfill_project_owner.py`)。
- **归属三风格并存**(散在 `services/auth/ownership.py` + `task_link_ownership.py`,非 DB 约束):
  1. 强 FK `owner_id`:仅 `projects`(创建时服务端强赋,列表按 `owner_id==current_user.id` 过滤)。
  2. 弱字符串 `created_by`(非 FK,空串=公共):`providers`;`models.created_by` 写了没用满。
  3. 无字段纯传递反查:chapters/shots/characters/files/generation_tasks。
- **公共资产坑**:`scene/prop/costume/actor` 的 `project_id` 可空,`NULL`=历史全局公共资产、任何登录用户可见。
- **无角色/权限模型**(纯"登录与否 + owner 匹配")。
- **测试**:354 全绿,用 `dependency_overrides[get_current_user]` 注入假 `User` + `AlwaysOwnedGetMixin`。

## 3. 已定产品决策(不再讨论,作为设计前提)

| # | 决策 | 说明 |
|---|---|---|
| D1 | **引入独立租户维度** | 一切业务实体归属到 tenant;user 属于 tenant |
| D2 | **同时支持 B2C + B2B** | B2C=注册即建"1 人租户"(tenant-of-one);B2B=一个租户多成员共享。共用同一套模型,个人用户只是成员数=1,不开特例 |
| D3 | **LLM 配置每租户各配一套** | 租户自带 provider/key,彼此不可见 → `providers`/`models` 成为带 `tenant_id` 的根实体 |
| D4 | **公共资产直接清理** | 删除 `project_id IS NULL` 的 scene/prop/costume/actor 行,字段转 NOT NULL,取消跨租户共享语义 |
| D5 | **认证外置、本期不改** | 未来接 OIDC 或其他第三方网关(方式未定);身份来源保持收敛在 `get_current_user` 单一出口,换认证只改这一处。本期不做 `/register`,继续 CLI 建号 |
| D6 | **本期收窄** | role 只落 `owner`/`member` 到模型(成员管理 API 后置);不做租户切换(单默认租户) |

## 4. 架构设计

### 4.1 数据模型
- **`tenants`**:`id String(64) PK`、`name`、`kind`(`personal`/`org`,仅元数据不分叉模型)、`TimestampMixin`。
- **`tenant_memberships`**:`id`、`tenant_id FK`、`user_id FK`、`role`(`owner`/`member`)、`status`,唯一约束 `(tenant_id, user_id)`。**这是"user 属于 tenant"的唯一权威**,天然支持一人多租户 / B2B 多成员。
- **`users.default_tenant_id`**:可空 FK,指向活跃租户的便捷冗余指针(可随时重算),降低每请求解析开销。membership 才是权威。
- **建号即建租户**:CLI `create_user`(及将来的注册)在同一事务里建 `Tenant(kind=personal)` + `membership(role=owner)` + 写 `default_tenant_id`。

### 4.2 归属字段收敛
- **只在"聚合根"加 `tenant_id`(NOT NULL, FK, index)**:`projects`、`scene`/`prop`/`costume`/`actor`/`characters`、`providers`、`models`。
- **子实体不逐表加**:chapters/shots/shot_details/各 image/link/file_usages/generation_tasks/task_links —— 只能经根到达,tenant 由根决定。
- **`owner_id`(projects)/`created_by`(providers/models)降级**为"租户内创建人/负责人"(审计展示用),**不再参与隔离过滤**;取消 `created_by` 空串=公共语义。
- **统一规则**(便于新人遵守):凡今天有 `assert_*_owned` 的实体,要么自带 `tenant_id`(根),要么只经带 `tenant_id` 的根到达。没有第三种。

### 4.3 隔离强制点(双层防御,默认安全)
**(a) 硬底座 —— session 级自动过滤:**
- 定义 `TenantScopedMixin`(带 `tenant_id` 列),所有根实体继承。
- 在 `core/db.py` 注册 `do_orm_execute` 事件:当 `session.info` 有 `tenant_id` 时,自动为所有 `TenantScopedMixin` 查询注入 `with_loader_criteria(TenantScopedMixin, lambda cls: cls.tenant_id == tid, include_aliases=True)`。→ 任何经该 session 的 SELECT(含懒加载/join)自动带租户过滤,**开发者忘写 where 也安全**。
- 注册 `before_flush`:对新增的 `TenantScopedMixin` 实例自动盖 `session.info["tenant_id"]`;对显式传入且不匹配的 tenant_id **抛错**(防越权写)。

**(b) 应用层 —— 收敛现有 asserts:**
- `ownership.py` 的 `assert_*_owned` **只重写函数体**为比较 `obj.tenant_id == current_tenant.id`,对外签名/404 行为不变 → 几十处调用点无需改。
- 删除 `assert_entity_owned` 的"`project_id` 为 None 放行"分支(公共资产已清理)。
- `task_link_ownership.py` 的多态反查复用重写后 asserts,自动获得租户语义。

**默认安全保证**:聚合层 `api/v1/__init__.py` 的 `dependencies` 从 `get_current_user` 升级为 `get_current_tenant`,请求进来就已在 session 盖好 tenant_id,新加路由天生被过滤。

### 4.4 身份注入(护住"换认证只改一处")
- **`get_current_user` 不动**(仍是唯一 token 解析出口,保证现有用法/测试兼容)。
- 新增 `get_current_tenant(current_user=Depends(get_current_user), db=Depends(get_db)) -> TenantContext`:
  - 从 `user.default_tenant_id`(回落唯一 membership)解析活跃租户,校验 membership 有效。
  - 产出轻量不可变 `TenantContext`(`tenant_id`/`role`/`user_id`),不直接返回 ORM。
  - **副作用**:写 `db.info["tenant_id"]` 供 4.3 的 ORM 事件读取。
- **换认证时**:未来外部网关注入身份/带 tenant claim,只改 `get_current_user`(读身份)+(若 token 带 tenant)`get_current_tenant` 解析分支;隔离底座、asserts、路由全不动。

### 4.5 非请求路径盲点 ⚠️
**Celery/后台任务**(`core/task_manager/stores.py`)运行在请求之外,不经 `get_current_tenant`。必须在任务入口**显式**从任务负载带出 tenant_id 并盖 `session.info["tenant_id"]`,否则后台写入漏盖 tenant 或被读过滤误伤。

## 5. 迁移策略(引入 Alembic)

因涉及"列转 NOT NULL、删存量行、加 FK/index",`create_all` 只建不改,单个幂等 CLI 也无法安全表达有序+可回滚的破坏性变更 → **就此引入 Alembic 承接 DDL**;数据删除/回填作为独立可审步骤(Alembic data migration 或配套 CLI,沿用项目习惯)。`create_all` 保留用于全新环境/测试快速建库,两者并存。

**严格有序的迁移步骤:**
1. 建 `tenants` + `tenant_memberships`(additive)。
2. 回填:每个存量 `User` 建 personal tenant + owner membership + 写 `default_tenant_id`(幂等 CLI)。
3. 根实体加 `tenant_id`(**先可空**);回填:`projects` 经 owner→membership→tenant;`characters`/四类资产经 project→tenant;`providers`/`models` 经 `created_by`→tenant(created_by 空串的历史 provider 归到开发者租户,不删)。
4. **删除** `scene/prop/costume/actor` 中 `project_id IS NULL` 的行及级联从属(先确认 `PRAGMA foreign_keys=ON`,SQLite 否则留孤儿)。
5. `tenant_id` 转 NOT NULL + FK + index;四类资产 `project_id` 转 NOT NULL。
6. 打开 4.3 的 ORM 强制 + 切 `api/v1` 门禁到 `get_current_tenant`。

**回滚考量**:第 4 步删除不可逆 → 执行前数据导出/备份,独立成相、独立审阅;加列/NOT NULL 转换有 Alembic downgrade 覆盖;建表可逆。
