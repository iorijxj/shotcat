# 对象存储访问加固方案（M4）

> 状态：方案定稿（2026-07-21），待按阶段执行
> 关联总纲：[公众化开发规划-20260720.md](公众化开发规划-20260720.md) M4（原名「对象存储签名 URL」）
> 依赖：无（独立工作流）；与 M2 多租户隔离对齐（已完成）

## 1. 背景与目标

shotcat 面向公众后，素材/生成图/视频存在对象存储（RustFS，S3 兼容）。**目标**：资产对象不再匿名公开可读，杜绝绕过后端的枚举/盗链，访问一律经后端代理并落到 M2 的租户归属校验。

## 2. 现状（事实基础，2026-07-21 摸底）

- **存储封装**：`app/core/storage.py`，boto3 + anyio 线程池；路径式寻址兼容 RustFS/MinIO。配置 `s3_*`（endpoint/bucket/key/`s3_base_path`/`s3_public_base_url`）。无 S3 配置时回落本地 `local-storage`（dev）。
- **前端访问**：资产 URL **已是后端代理** `/api/v1/studio/files/{file_id}/download`（`entity_thumbnails.py` 生成），该下载路由**已有 `assert_file_owned` 租户鉴权**（`files.py:103`，P4c 起按租户 404）。代理读取整对象转流（`storage.download_file`）。
- **真实暴露洞（本方案要堵）**：
  1. 上传对象设 `ACL=public-read`（`services/studio/files.py:168`、`utils/files.py:126`）→ 对象在 bucket 里**匿名公开可读**。
  2. 对象 key = `files/{原始文件名}`（`files.py:163`）→ **可预测、可枚举、可覆盖**。
  3. 二者叠加 → 攻击者猜到/枚举 key 即可**直连 bucket 下载，完全绕过 `assert_file_owned`**，代理鉴权形同虚设。
- **FileItem.thumbnail** 存的是 `info.url`（`_build_public_url` 生成的 **bucket 公开 URL**），bucket 私有后该 URL 失效（实体图列表另经 `resolve_thumbnails` 返回代理 URL，两者不一致）。
- **无任何签名/presign 代码**。
- **租户关系**：`FileItem` 无 `tenant_id`（子实体），经 `FileUsage.project_id → Project.tenant_id` 归属；代理下载的 `assert_file_owned` 已按此校验（P4c 租户化）。

## 3. 已定决策

| # | 决策 | 说明 |
|---|---|---|
| D1 | **代理 + bucket 私有**（不做前端直连签名 URL） | 下载已是代理且已租户鉴权；把对象私有化即彻底堵洞。前端直连签名 URL 是带宽优化、非安全必需，后置。 |
| D2 | **上传 key 不可预测** | `files/{uuid}{ext}`（保留扩展名供下载名/Content-Type），消除枚举/覆盖。 |
| D3 | **停设 public-read + bucket 私有化** | 代码不再传 `ACL=public-read`；bucket 匿名读策略由运维关闭（RustFS/MinIO policy），`init_storage` 不建公开策略。 |
| D4 | **thumbnail 存代理 URL** | `FileItem.thumbnail` 改存 `/api/v1/studio/files/{id}/download`，与实体图一致；存量行回填。 |
| D5 | **`s3_public_base_url`/`_build_public_url` 收窄** | 仅保留给 local-storage dev；S3 私有模式不再对外发 bucket URL。 |
| D6 | **不改租户校验** | 代理下载复用 M2 的 `assert_file_owned`，本方案不新增隔离逻辑。 |

## 4. 架构设计

- **上传**：`upload_file` 生成 `files/{uuid}{ext}` key、**不传 ACL**、`thumbnail` 写代理 URL。存储层默认不公开。
- **下载**：维持现状代理（`download_file_api` → `assert_file_owned` → `build_download_response` 用带凭证 client 读私有对象转流）。对象私有后，绕过代理直连 bucket 返回 403。
- **私有化**：`init_storage` 建桶不设匿名策略；运维在 RustFS 侧确保 bucket policy 无 anonymous read。代码侧移除所有 `public-read`。
- **存量兼容**：旧对象 key 仍可预测但 bucket 私有后不可直连，无需重命名（非破坏性）；仅需回填 `thumbnail` 为代理 URL，否则私有化后前端旧图挂。

## 5. 分阶段实施（每阶段独立可验、独立提交）

| 阶段 | 内容 | 验收 |
|---|---|---|
| **M4-1 上传加固** | key 改 `files/{uuid}{ext}`；移除所有 `ACL=public-read`（`services/studio/files.py`、`utils/files.py`、worker 上传点若有）；`thumbnail` 存代理 URL | 上传后 `storage_key` 不含原始文件名、无 public-read；`thumbnail` 是 `/api/v1/studio/files/{id}/download`；测试全绿 |
| **M4-2 存量 thumbnail 回填** | 幂等 CLI `backfill_file_thumbnails`：把 `FileItem.thumbnail` 从 bucket URL 重写为代理 URL | 重复执行幂等；回填后旧文件详情/列表 thumbnail 为代理路径 |
| **M4-3 bucket 私有化** | `init_storage` 不建公开策略；`_build_public_url`/`s3_public_base_url` 收窄注释为 dev-only；文档化运维私有化步骤 | 直连 bucket 公开 URL 返回 403/AccessDenied；经代理下载正常；测试全绿 |
| **M4-4（后置，可选）** | 大文件（视频）后端签发短期 presigned GET 卸载带宽 | —（本期不做，明确记录） |

**上线运维序**：部署 M4-1（新上传即私有 key）→ 跑 M4-2 回填存量 thumbnail → 运维关 bucket 匿名读（M4-3）。顺序不可颠倒（先回填再私有化，否则旧图挂）。

## 6. 必改文件清单

- `app/core/storage.py` —— 私有化（`init_storage` 不设公开）、`_build_public_url` 收窄 dev-only、上传不默认公开
- `app/services/studio/files.py` —— `upload_file`：UUID key、去 ACL、thumbnail 代理 URL
- `app/utils/files.py` —— 去 `public-read`
- worker 图片/视频上传点（实施时 grep `public-read`/`upload_file` 确认）
- 新增 `app/cli/backfill_file_thumbnails.py` —— 存量 thumbnail 回填
- `app/config.py` —— `s3_public_base_url` 注释更新为 dev-only 语义
- 总纲 `公众化开发规划-20260720.md` §1 —— M4 子文档链接指向本文

## 7. 遗留与风险

- **回填与私有化的顺序依赖**（先回填 thumbnail 再关公开），运维序已固化。
- **local-storage dev 模式**（无 S3）保持本地公开路径，不受影响。
- 存量对象 key 仍可预测——私有化后不可直连即安全，不重命名（非破坏性，避免动存量数据）。
- **签名 URL（M4-4）**：性能优化，本期不做；若后续视频带宽成为瓶颈再评估，届时需 bucket 已私有 + 签发前校验租户。
