## 阶段二：API Key 暴露修复（本次体检重点，明确优先于其余功能性风险）

### 2.1 现状确认（2026-07-19 复核结论）

- `Provider.api_key` / `api_secret`：`app/backend/app/models/llm.py:78-79`，**明文存储**，`String(4096)`，无加密。
- `ProviderRead` 响应 schema（`app/backend/app/schemas/llm.py:51-56`）**不包含** api_key/api_secret 字段——GET 接口不会把明文 key 回显给前端，这点是好的，不用重做。
- **真正的洞在这里**：`app/backend/app/api/v1/__init__.py` 里 `/llm` 路由只挂了"必须登录"（`dependencies=[Depends(get_current_user)]`），**没有所有者校验**。`app/backend/app/services/llm/manage.py` 的 `get_provider`/`update_provider`/`delete_provider`/`list_providers_paginated` 全部不区分 `created_by`。也就是说：**任何登录用户都能修改/删除任何其他用户创建的 Provider**，包括把 `base_url` 改成攻击者自己的服务器地址——下一次任何用户用这个 Provider 生成文本/图片/视频时，后端会把存量的**真实明文 api_key** 当 `Authorization: Bearer <key>` 请求头发到攻击者的服务器上（`app/backend/app/dependencies.py:56-92` `_ImageHttpRunnable` 的逻辑），相当于**用改 base_url 的方式把别人的密钥钓出来**。这是一条比"直接读密钥"更隐蔽、但同样严重的攻击链。
- `created_by` 字段目前由请求体客户端自己填（`ProviderCreate.created_by`，`app/backend/app/schemas/llm.py:24`），不可信，和 `Project.owner_id` 此前的问题一模一样。
- `cryptography>=42.0.0` 已经是现有依赖（`pyproject.toml`），但**全仓库没有任何地方真正用它加密过东西**——有依赖没实现。

### 2.2 修复任务（建议顺序）—— 2026-07-19 已全部完成

| # | 任务 | 说明 | 状态 |
|---|---|---|---|
| 2.1 | Provider/Model 加真实归属：`Provider.created_by` 改服务端从 `current_user.id` 赋值，`ProviderCreate`/`ModelCreate` schema 去掉可写的 `created_by` | `app/services/auth/ownership.py` 新增 `assert_provider_owned`/`assert_model_owned`（Model 经 `provider_id` 两跳反查，同 Chapter→Project 模式）；`app/api/v1/routes/llm.py` 全部 Provider/Model 端点加校验；空 `created_by` 视为迁移期公共资源（呼应存量数据） | ☑ |
| 2.2 | 堵住"改 base_url 钓密钥"这条链 | 2.1 做完后自然收口；新增 `app/backend/tests/test_provider_ownership.py`，含专门断言"用户 B 改不动用户 A 的 provider.base_url"的回归用例 | ☑ |
| 2.3 | `api_key`/`api_secret` 数据库层加密 | 新增 `app/core/secret_crypto.py`（Fernet），`app/models/llm.py` 新增 `EncryptedSecret` TypeDecorator 做透明加解密（对存量明文解密失败时原样透传，兼容迁移前数据）；列类型从 `VARCHAR(4096)` 改 `TEXT`（迁移 `sql/011-widen-provider-secret-columns.sql`）；密钥走 `PROVIDER_SECRET_ENC_KEY` 环境变量（必填，`.env.example` 留占位）；一次性迁移脚本 `app/cli/migrate_provider_secrets.py`（读出触发透传/解密、原样重新赋值触发加密写回，可重复执行） | ☑ |
| 2.4 | 检查日志/异常信息有没有把 api_key 打出来 | 复核 `app/services/worker/task_logging.py` 及全部调用点（`image_task_runner.py`/`generated_video.py`/`shot_frame_prompt_tasks.py`），只传显式字符串字段（task_id/event/stage/error），没有任何调用点把整个 `run_args`/`payload` 字典打进日志，未发现泄露路径 | ☑ |

**已知残留风险（本轮明确不修，需求方已确认范围）**：图片/视频/提示词生成任务会把**解密后的明文 api_key** 复制一份写进 `generation_tasks.payload`（`app/backend/app/services/film/generated_video.py:180`、`app/backend/app/services/studio/image_task_runner.py:270` 等处的 `run_args` 构建逻辑），这份明文副本长期留存、不会清理，且 2.3 的字段加密**不覆盖**这里——加密只管住了 `providers` 表本身。好在目前没有任何 API 接口会把 `payload` 字段返回给前端（`/tasks/{id}/status`、`/result` 只返回 `status`/`result`/`error`），所以不通过接口泄露，只有直接连数据库才能看到。彻底解决需要把 `run_args` 改成只存 `provider_id`/`model_id`，等 worker 真正执行时再实时解密取 key，这属于对异步任务执行链路的改造，风险和工作量都明显更大，本轮不做，留作后续独立评估。

> 阶段二已完成（2026-07-19）：299 个测试全绿（含 `test_provider_ownership.py` 6 个跨用户越权用例、`test_secret_crypto.py` 4 个加密用例）。**"登录用户之间互相偷密钥"这条链已经堵上**（针对 providers 表本身；上面提到的 run_args 残留风险除外）。

## 明确不做 / 暂不需要的事项（避免过度工程）

- **不**启动"正规改底座"（33 表全量加 owner + 123 接口逐个改造，工期 50-77 人天）——当前"应急隔离 + 定向鉴权补丁"的组合已经解决了实际被提出的问题，规模不到需要动上游源码的程度。
- **不**对 Scene/Prop/Costume/Actor 四类允许 `project_id` 为空的历史资产收紧为"必须归属"——这是产品既有设计（跨项目复用资产），不是漏洞，之前已跟需求方确认过。
- **不**给 `ModelSettings`（全局单例设置）加归属——它本来就该是全局共享的默认值，除非未来做管理员角色，否则不需要按用户区分。
