# shotcat 前端收敛方案：以 web/ 为唯一日常入口（供确认）

> 版本：v2　|　日期：2026-07-19　|　状态：已确认并落地
> 触发：上游 commit `794807a`（docs: 明确当前工作台入口与本地数据边界）已合并入本仓库（合并 commit `36052b2`）。该 commit 在文档层面正式把 `app/front/` 降级为"旧 Studio 管理界面，仅维护/历史兼容"，把 `web/` 定为"当前产品界面、功能验证与反馈的唯一基准"。
> 目标：把这句话从"文档层面的说法"落实到"部署与开发脚本的行为"，消除仓库里 4 个 `.bat` 脚本对"谁是主前端"认知不一致的问题。

---

## 一、现状盘点（为什么两个前端会打架）

仓库里现在确实是两个独立前端：

| | `web/` | `app/front/` |
|---|---|---|
| 技术栈 | React 18 + Vite + react-router，手写 `lib/api.ts` | 平台原生 Studio，Ant Design + OpenAPI 生成客户端 |
| 端口（dev / server.bat） | 5273 / 18081 | 7788 / 18080 |
| 定位（合并后 README 已明确） | 日常制作唯一入口 | 旧 Studio 管理界面，仅维护与历史兼容 |

用户给的截图（标题"shotcat 短剧工作台"，"作品库"页，"新建剧本项目"按钮，右上角猫头像）经核对就是仓库现有的 `web/src/pages/Lobby.tsx`——也就是说**它不是新东西，是 `master` 上已经存在的 `web/`**，上游那次更新只是在文档里正式给它"转正"、把 `app/front/` 摁下去，并没有推送新代码。

但仓库里 4 个 `.bat` 入口脚本对"谁是主角"互相矛盾：

| 脚本 | 用途 | 现在默认起的前端 | 知道 `web/` 存在？ |
|---|---|---|---|
| `install.bat` | 首次装依赖 + 预构建镜像 | 只装 `app/front`（264-276 行） | 否 |
| `run.bat` | 快速启动（假设已装好） | 只起 `app/front`（15/21/71-76 行） | 否 |
| `test.bat` | 改代码后全量重装+起服务 | 只起 `app/front`，还带 `openapi:update`（15/78-116 行） | 否 |
| `server.bat` | 无头/局域网部署 | **同时**起 `web/`（主）和 `app/front`（次），两者都对外暴露（56-68 / 151-169 / 216-217 行） | 是，但没"屏蔽"旧前端 |

结论：本地开发三件套（install/run/test）完全没接入 `web/`，日常开发者跑这三个脚本根本看不到截图里的页面，只会看到 `app/front` 的旧管理界面；只有最新写的 `server.bat` 知道 `web/` 是主角，但它仍然把 `app/front` 一起摆上台面、一起开防火墙端口暴露到局域网。

另外，`CLAUDE.md` 的"三方定位"表（`app/`、`knowledge/`、`_archive/screenwriter/`、`bridge/`）从建仓起（`fc912b7`）就没提过 `web/`——这不是本次改动引入的新遗漏，是一直存在的盲区，这次一并补上。

## 二、上游做了什么、我们要跟进做什么

上游 `794807a` 的动作 = **文档层面单向度确认**："以后只认 `web/`，`app/front` 别再当成产品对照基准"。它没有改任何代码或部署脚本（详见上一轮分析：改动的 6 个文件全是 README / docs）。

我们要做的 = 把这句话在部署与开发脚本里落实：本地默认只起 `web/`，局域网部署默认只对外暴露 `web/`，`app/front` 保留但退到"需要时手动维护"的位置。这就是用户所说的"屏蔽一个前端"。

## 三、改动清单

### 1. `CLAUDE.md` —— 补第四方定位

在"三方定位（勿混淆）"里加一条：

```
- `web/` = 日常制作入口，唯一对外/对用户的产品界面（React+Vite，手写 API 封装，端口 5273）。今后功能开发以此为准。
```

并在 `app/` 那条后面补一句：`app/front/`（Studio 原生前端）现降级为历史遗留维护通道，不再是产品界面，不接新需求。

### 2. `install.bat` —— 装依赖时把 `web/` 也覆盖

- 264-276 行目前只 `pnpm install` `app/front`、只预构建 `backend celery-worker front` 镜像。
- 加一步 `pnpm --dir web install`（`web/` 是纯前端 vite 项目，不需要 docker 镜像，不用改 compose build 那行）。
- `app/front` 的 `pnpm install` + docker 镜像预构建**保留**——它仍是要跑起来的遗留服务，只是不再"首要"。

### 3. `run.bat` / `test.bat` —— 默认起 `web/`，`app/front` 退为可选

- 两个脚本目前硬编码只起 `FRONT_DIR=app/front`（7788）。
- 改为默认起 `web/`（`pnpm dev`，5273），提示语从 `frontend` 改成 `workbench (web/)`。
- `app/front` 的 dev server 改为**不默认启动**：脚本打印一行提示"如需维护旧 Studio 管理界面，另开终端 `cd app/front && pnpm dev`"，而不是自动开窗口。
- `test.bat` 里的 `pnpm run openapi:update`（105-108 行）是 `app/front` 自己的 OpenAPI 客户端生成步骤，`web/` 用手写 `api.ts` 不需要它。`app/front` 默认不再启动后，这步也应只在显式维护 `app/front` 时才跑，日常 `test.bat` 可以跳过、加快启动速度。

### 4. `server.bat` —— 局域网部署不再对外暴露 `app/front`

这是"屏蔽"的核心落地点：

- 137 行防火墙循环 `for %%P in (!SERVER_FRONT_PORT! ...)`：去掉 `SERVER_FRONT_PORT`，只留 `SERVER_BACKEND_PORT` 和 `SERVER_WEB_PORT`。
- 151-153 行 Caddyfile 里 `:!SERVER_FRONT_PORT! { reverse_proxy ... }` 整块删除。
- 207 行 self-check 探测列表里的 `SERVER_FRONT_INTERNAL_PORT` 去掉（容器仍在内部跑，但不再对外验收）。
- 217 行启动完成提示 `Studio admin (app/front): http://localhost:%SERVER_FRONT_PORT%` 删除，或改成"仅内部访问，未对外暴露"。
- `front` 容器本身是否还要随 `docker compose up -d --build` 一起构建启动——见第四节待确认项 1。

### 5.（可选）`docker-compose.yml` —— `front` 服务加 profile

如果决定"连容器都不常驻"，给 `front` 服务加：

```yaml
front:
  profiles: ["legacy"]
  ...
```

默认 `docker compose up -d` 会跳过它，需要维护时手动 `docker compose --profile legacy up -d front`。这样 `install.bat` 的预构建列表、`server.bat` 的 `up -d --build` 都不用改（profile 自动排除）；`run.bat`/`test.bat` 里 `docker compose stop backend front celery-worker` 对不存在的容器是安全空操作，也不用改。

### 6. `.env.example` —— 注释更新

`SERVER_FRONT_PORT` / `SERVER_FRONT_INTERNAL_PORT` 两行注释加一句："`app/front` 现为维护用途，默认不再通过 Caddy 对外暴露；如需临时访问，用 SSH 端口转发到该内部端口。"

### 7. `README.md` —— 补一句免责

"开发与验证"一节（113-118 / 129-132 行）`cd app/front && pnpm run typecheck` / `openapi:update` 前加一句："以下两步只在维护 `app/front` 旧 Studio 界面时需要，日常基于 `web/` 的开发不需要执行。"

### 8. `docs/99_归档/应急上线安全方案.md` —— 顺带简化（非阻塞）

P0 第 2 条"给后端与前端站点加 basicauth"里的"前端"目前隐含两个站点；`app/front` 不再对外暴露后，只需要给 backend + `web/` 两处加口令，工作量略减。可在下次修订该文档时更新，不必现在动。

## 四、待确认项

1. **`app/front` 容器要不要彻底不随 `docker compose up` 常驻？**
   - 方案 A（推荐）：容器继续内部常驻，只是不再对外暴露端口——改动小、可逆，维护时随时能从内部访问。
   - 方案 B：加 `profiles: legacy`，默认完全不启动，需要时手动拉起——更彻底"屏蔽"、省资源，但维护前多一步手动操作，且要同步改 `install.bat` 的预构建逻辑。
2. **`run.bat` / `test.bat` 默认要不要保留"同时起两个前端"？**
   - 方案 A（推荐）：默认只起 `web/`，`app/front` 打印手动命令、不占用窗口。
   - 方案 B：默认两个都起（跟 `server.bat` 现状一致），只是标签区分清楚谁主谁次。
3. 是否现在就动 `docker-compose.yml` 的 profile（有一点复杂度），还是先只做 `server.bat` 的对外端口收敛（改动小、立即见效，profile 留作二期）？

以上 3 点确认后即可按第三节清单逐项动手改脚本与文档。

## 五、落地记录（2026-07-19）

已确认选型：

1. `app/front` 容器不随 `docker compose up` 常驻 —— 采用方案 B。
2. `run.bat`/`test.bat`/`install.bat` 只以 `web/` 为唯一前端，不再默认起 `app/front` —— 采用方案 A（严格版：连 dev 依赖安装都不再默认覆盖 `app/front`，需要时手动维护）。
