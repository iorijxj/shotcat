# 短剧工作台 · 流程优化研究（Opus 统筹）

**研究对象**：猫叔的短剧工作台（web :5273 ＋ Jellyfish 后端 :8000 ＋ bridge pipeline :5280）
**流程链**：新建项目 → 剧本 → 抽取设定 → 锁定视觉词典 → 造型图 → AI 拆镜头 → 行内编辑 → 视听单元 → 首/关/尾帧 →（图生视频）
**实验规范**：每次实验先写目标再动手；记录方法、实际结果（数据）、六问评估；发现问题当场修正并记录修正内容；测试项目一律用 `proj_lab` 前缀，结束清理。

## 实验记录

> 数据来源：只读项目 proj_echo《回声》(679字/31镜)、proj_test95b8《夜班出租》(244字/14镜) 的落库数据 + 代码链路追踪 + 新建 proj_lab* 对照实验。

### 实验 1：新建项目表单三字段（style / visual_style / 画幅）对下游的真实影响
- **目标**：追踪 题材风格/画面表现/画幅比例 三字段是否真正流向生产，还是摆设。
- **这步骤为什么存在**：项目级全局参数，理应统一约束下游造型与出图。
- **方法**：从 web/Lobby 表单 → api.ts → 后端逐层 grep 追踪三字段消费点。
- **实际结果**：
  - `default_video_ratio`（画幅）**有真实效果**：Frames.tsx/Storyboard.tsx 出图时传 `target_ratio` → 后端 `resolve_image_size`（image_capabilities.py）映射为真实像素（9:16→1024x1792、16:9→1792x1024）。画幅直接决定成图尺寸。
  - `style`（题材）+`visual_style`（画面表现）**有真实效果**：extract_setup 把二者写进每个实体；帧提示词 build_run_args 从 project 读取 visual_style/style/unify_style，注入 ShotFramePromptAgent（agents 第120-121行"必须继承项目级画面表现形式与题材风格：{visual_style}/{style}"）。
  - **缺口**：visual_dict.py 与 shot_breakdown.py **完全不读** style/visual_style/画幅——视觉词典和分镜阶段对"动漫 vs 现实""竖屏 vs 横屏"无感知；直到出图才生效。竖屏短剧的构图（9:16）在分镜/词典阶段没有任何竖构图约束。
  - **小 bug**：api.ts `createFrameImageTask` 默认参 `'16:9'`，Frames.tsx 兜底也是 `'16:9'`，而 Storyboard/Overview 兜底是 `'9:16'`。project 正常加载时无碍（都传真实值），但 project 未就绪时 Frames 会用 16:9 兜底，与短剧默认竖屏不一致。
- **改善空间**：把画幅（尤其竖屏构图约束）下沉到 shot_breakdown 与帧提示词；把 Frames 兜底改成 9:16。
- **对新手**：字段有效但"画面表现=动漫/现实"只有两项，且分镜看不到差异，新手不易理解它到底改了什么。
- **对老手**：老手需要的"分镜级画幅覆盖"后端有 `override_video_ratio` 字段，但 UI 完全没暴露。
- **是否优秀产品**：字段非摆设合格；但生效链路太靠后、无中途反馈，及格偏下。
- **当场修正**：见实验末尾"待决策"，画幅下沉属结构改动，记入待决策。

### 实验 2：运镜多样性——夜班出租 14 镜全 STATIC
- **目标**：核实"运镜全固定机位"问题，定位是 prompt 还是模型问题。
- **这步骤为什么存在**：运镜（推拉摇移）是分镜的核心叙事语言，全固定=PPT。
- **方法**：统计两项目 movement 分布；对比 shot_breakdown.py 提示词。
- **实际结果**：夜班出租 movement 全 14/14 STATIC；回声 31 镜中 STATIC 19、另有 PAN3/DOLLY_IN3/TRACK2/TILT1/ZOOM_IN1/DOLLY_OUT1/HANDHELD1（多样）。两者同一套 SYS 提示词，差异主要来自剧本（夜班出租静态对话戏多）+ GLM 保守倾向。提示词里"情绪爆发可推镜"是**软建议**，无硬约束，GLM 遇到平淡对话戏时一律 STATIC 偷懒。
- **改善空间**：在 SYS 加硬约束——"全片 STATIC 占比不得超过 X%，进入新场景的建立镜头优先用运动镜头交代空间"。(下见实验19 A/B验证)
- **对新手**：新手看不出"全固定"是问题，会照单全收 → 成片呆板。
- **对老手**：老手会逐条改运镜，等于分镜白拆，反而增加工作量。
- **是否优秀产品**：这是当前分镜最弱环节之一，不合格。
- **当场修正**：留待实验 19 做 prompt A/B 再定稿。

### 实验 3：道具抽取偏多——夜班出租 8 个道具
- **目标**：核实 extract-setup 道具过抽取，测试"只留关键道具"约束是否有效。
- **这步骤为什么存在**：道具实体用于后续锁定外观+一致性，过多则造型工作量爆炸、且多为环境装饰。
- **方法**：看夜班出租抽出的 8 道具。
- **实际结果**：8 个 = 后视镜/挡风玻璃/方向盘/车窗/计价器/纸巾/旧纸箱/居民楼。其中真正有叙事功能的只有：旧纸箱（林小雨的家当，情感载体）、纸巾（陈默递出的善意）、计价器（"开到天亮"的钱意味）。其余 5 个（后视镜/挡风玻璃/方向盘/车窗/居民楼）是场景固有构件或背景，不该单列为道具实体。过抽取率 5/8≈62%。SYS 虽写"纯装饰不算"，但对"车的构件算不算道具"无判据，GLM 把镜头里出现的物体都抽了。
- **改善空间**：prompt 加判据"可被角色拿起/带走/递出/操作、或被台词点名、或推动情节的才算道具；车/房/门窗/家具等固定构件归场景，不单列"。(实验18 A/B验证)
- **对新手**：新手会对着 8 个道具逐个生成造型图（花钱），大量浪费。
- **对老手**：老手需手动删一半，不如不抽。
- **是否优秀产品**：不合格，直接影响成本与体验。
- **当场修正**：留待实验 18 prompt A/B 定稿。

### 实验 4：机位角度单一——EYE_LEVEL 主导
- **目标**：检查 angle 分布是否也存在单一化。
- **方法**：统计两项目 angle 分布。
- **实际结果**：夜班出租 13/14 EYE_LEVEL（仅计价器 1 个 HIGH_ANGLE）；回声 25/31 EYE_LEVEL（HIGH3/LOW2/OVER_SHOULDER1）。与运镜同病：提示词只在"对话用过肩"给软建议，无硬约束，GLM 默认平视。夜班出租多组对话戏却零 OVER_SHOULDER（本该正反打），说明"对话用过肩+正反打"这条软建议基本没被执行。
- **改善空间**：把"双人对话必须至少一组过肩正反打"升级为硬约束。
- **对新手**：平视一镜到底，视觉平淡，新手无从判断。
- **对老手**：老手期待的正反打没出，需重排。
- **是否优秀产品**：与实验 2 同因，分镜语言贫乏，不合格。
- **当场修正**：并入实验 19 运镜/机位 A/B。

### 实验 5：空镜"空镜："前缀规则遵守度
- **目标**：验证 SYS 里"无人物空镜必须以'空镜：'开头"是否被 GLM 执行。
- **这步骤为什么存在**：空镜前缀是给下游帧提示词/图像模型的信号——后端 `_build_character_context` 对空镜会强制"画面不出现任何人物"，防止图像模型脑补路人。前缀是这条链的触发词之一。
- **方法**：夜班出租 characters=[] 的 3 个镜头（挡风玻璃雨点/霓虹积水/尾灯驶离）检查 action 开头。
- **实际结果**：**规则未被遵守**。3 个空镜的 action 分别是"雨点开始砸在挡风玻璃上…""出租车停在路边，霓虹灯…""出租车尾灯划破雨幕…"，**无一以"空镜："开头**。GLM 忽略了这条前缀指令。（注：后端空镜保护实际靠 characters 数组是否为空判断，不完全依赖前缀，所以危害有限，但提示词里的规则形同虚设，说明"埋在长 SYS 里的格式指令"GLM 遵守率低。）
- **改善空间**：要么删掉这条没用的规则，要么把空镜判定与前缀改为后处理（characters 为空时自动加前缀），不依赖 GLM。
- **对新手**：无感知（后端有兜底）。
- **对老手**：老手若依赖前缀做筛选会踩空。
- **是否优秀产品**：小瑕疵，规则与实现脱节。
- **当场修正**：改为后处理自动补前缀（见下）。

### 实验 6：角色/场景名称原样保留（铁律核查）
- **目标**：验证全链路是否违反"名称原样保留"铁律。
- **方法**：核对 extract 抽出的实体名 vs 剧本；shot 的 scene 字段 vs 场景实体名。
- **实际结果**：**完全合规**。夜班出租角色 陈默/林小雨、场景 出租车内-夜/街头-夜，与剧本原文一致；14 个镜头的 scene 字段全部命中已有场景名（shot_breakdown 用 scene_id_by_name 映射，未命中则 scene_id 留空，不会造脏名）。回声同样合规。代码层面：extract 的 alloc() 用"原名命中复用旧 id"，visual_dict 用名称做 JSON 键回填、未匹配则跳过并告警（不改名）。铁律有代码保障。
- **改善空间**：visual_dict 名称未匹配时目前只跳过告警，可加"模糊匹配提示"帮定位 GLM 改名。
- **对新手/老手**：一致性是底座根基，两类用户都受益。
- **是否优秀产品**：这条做得好，是产品的正确根基。
- **当场修正**：无。

### 实验 7：镜头数与剧本长度的比例规律
- **目标**：找"每百字出几镜"的经验规律，判断分镜密度是否合理。
- **方法**：两项目对比。
- **实际结果**：回声 679字→31镜（4.56 镜/百字，单场景）；夜班出租 244字→14镜（5.74 镜/百字，2 场景）。短剧经验值约 5 镜/百字。夜班出租每场景 7 镜（出租车内）与 7 镜（街头），符合 SYS"短片每场景 3-6 个镜头"上限附近。密度本身合理，问题不在数量在多样性（见实验2/4）。
- **改善空间**：分镜密度可作为项目级可调参（快节奏/慢节奏短剧不同）。
- **对新手**：默认密度合理，新手不用操心。
- **对老手**：老手可能想控密度，目前不可调。
- **是否优秀产品**：数量维度合格。
- **当场修正**：无。

### 实验 8：对白时长换算规则的边界
- **目标**：验证"有对白镜头 duration≥对白字数÷4"是否落实，边界是否合理。
- **这步骤为什么存在**：镜头时长必须容纳台词朗读时间，否则图生视频卡不下台词。
- **方法**：核对夜班出租 5 个对白镜头的字数与 duration；读 `_dlg_min_secs`/`_duration` 代码。
- **实际结果**：规则**有代码强制**（去标点计字，(n+3)//4 向上取整，与 GLM 建议取 max，夹 1-60）。核对：陈默长台词"我拉过三百个…最后都到家了"(18字→5s下限) duration=7 ✓；"随便开。开到天亮"(8字→2s) =4 ✓；"坐稳。我带你去看日出"(10字→3s) =4 ✓。全部满足下限。边界：慢速抒情戏按每秒3字更稳，代码注释提及但 `_dlg_min_secs` 硬编码每秒4字，未按情绪分档。前端 Storyboard 还有"说不完⚠"实时校验（dc>0 且 duration<需求）。
- **改善空间**：长独白（>40字）可再加"预留呼吸停顿"余量；抒情戏按 3 字/秒。
- **对新手**：自动兜底+前端警示，新手不会做出"台词说不完"的镜头，很贴心。
- **对老手**：老手手改时前端即时校验，体验好。
- **是否优秀产品**：这条是亮点，代码+UI 双保险。
- **当场修正**：无。

### 实验 9：extract-setup 道具约束 prompt A/B（对照实验 + 已修）
- **目标**：验证"只留影响剧情的关键道具"约束能否解决道具过抽取。
- **方法**：同一剧本灌入两个 lab 项目，A=原 SYS，B=加判据+固定构件排除清单的 SYS，比对道具数。另用夜班出租剧本（已知基线 8 道具）跑 B。
- **实际结果**：
  - 办公室剧本（315字，道具本就不多）：A=4（旧怀表/辞职信/钢笔/银色相框），B=3（去掉银色相框）。
  - 出租车剧本（夜班出租原文）：基线 A=8（后视镜/挡风玻璃/方向盘/车窗/居民楼/计价器/旧纸箱/纸巾），约束 B=**2**（旧纸箱/纸巾）。B 正确剔除全部 5 个固定构件，过抽取率从 62% 降到 0。
  - 副作用：B 把 计价器 也剔了（"打表"其实有情节意味，属过度收敛的边界项）；总体宁缺毋滥可接受，用户可手动补。
  - 耗时：extract 单次 48-106s（glm-4.6，244-315字），主要花在 GLM。
- **改善空间**：边界项（计价器/相框这类"操作型/象征型"）可让 GLM 输出置信度或标"可选"，交用户勾选。
- **对新手**：直接少造一半没用的造型图，省钱省心，价值大。
- **对老手**：省去手动删道具。
- **是否优秀产品**：修正后合格，是本轮性价比最高的改动之一。
- **当场修正**：**已把约束写入 extract_setup.py SYS**（含固定构件排除清单），py_compile 通过。

### 实验 10：抽取后即时读取的最终一致性缺口 + 场景名跨 run 不稳定
- **目标**：附带发现——校验数据可靠性与命名稳定性。
- **方法**：extract 返回后立刻 GET 实体；对比两次 run 的场景名。
- **实际结果**：
  - **最终一致性缺口**：baseline 跑完立刻查 character 只返回 1 个（林晚），漏了最后创建的 顾言（带服装，创建最晚）；延时 2s 再查即为 2 个。根因是 平台 的 commit-after-yield（HTTP 响应发出后才提交），pipeline 内部用 wait_get 处理了依赖，但**外部读取者（前端/别的脚本）在某步刚结束时会读到陈旧数据**。前端"抽取完成后刷新实体列表"若不延时/重试，可能少显示实体。
  - **场景名不稳定**：夜班出租基线抽出"出租车内 - 夜""街头 - 夜"，约束 run 抽出"出租车内""街头"（丢了" - 夜"）。场景名是 GLM 每次即兴生成，跨 run 不一致。这不违反"名称原样保留"（那条针对已定实体），但意味着**重跑 extract 会产生新场景名 → 新实体 id**（alloc 按原名复用，新名=新增），可能留下孤儿场景。
- **改善空间**：① 前端 extract 完成后刷新实体列表加 1-2s 延时或轮询重试；② extract 提示 GLM 场景名用"地点"不带时间后缀，或重跑时做名称模糊归并。
- **对新手**：新手重跑抽取可能看到场景变多（新旧名并存），困惑。
- **对老手**：老手会发现 id 漂移。
- **是否优秀产品**：一致性缺口是底座固有时序问题，前端未完全防御，中等风险。
- **当场修正**：暂记待决策（涉及前端刷新逻辑与 extract 重跑语义）。

### 实验 11：shot-breakdown 运镜/机位多样性硬约束 A/B（对照 + 已修）
- **目标**：验证加"硬约束"能否解决运镜全 STATIC、机位全 EYE_LEVEL、对话无过肩的问题。
- **方法**：夜班出租剧本，基线=proj_test95b8（现网），约束版=proj_lab_taxi 用追加 5 条硬约束的 SYS。对比 movement/angle 分布。
- **实际结果**（16 镜 vs 基线 14 镜）：
  | 指标 | 基线 | 约束版 |
  |---|---|---|
  | STATIC 占比 | 100% (14/14) | **25% (4/16)** |
  | EYE_LEVEL 占比 | 93% (13/14) | **37.5% (6/16)** |
  | 过肩正反打 | 0 | **4 个 OVER_SHOULDER**（成对反打） |
  | 运镜种类 | 1 (STATIC) | **7** (DOLLY_IN×4/PAN×2/ZOOM_IN×2/TRACK×2/STATIC×4/CRANE/DOLLY_OUT) |
  | 机位种类 | 2 | **6** (EYE/OVER_SHOULDER/HIGH/LOW/BIRD_EYE/DUTCH) |
  - 两个新场景的建立镜头（镜1 出租车内=DOLLY_IN、镜8 街头=CRANE）均为运动镜头，满足约束2。
  - 对话戏出现过肩正反打（约束3 生效）。
  - 耗时 166s（比基线略长，因需先规划配比；仍可接受）。
- **改善空间**：16 镜略多于 14（运动建立镜增加），可再限制总镜数；DUTCH/BIRD_EYE 偶发可能偏炫技，可标"仅情绪失衡时用"。
- **对新手**：一键得到有电影感的分镜，无需懂镜头语言，价值极大。
- **对老手**：省去逐镜补运镜/正反打，接近可用初稿。
- **是否优秀产品**：修正前是最弱环节，修正后跃升为亮点。
- **当场修正**：**已把 5 条硬约束写入 shot_breakdown.py SYS**，py_compile 通过。

### 实验 12：空镜"空镜："前缀后处理修复验证
- **目标**：验证实验 5 的后处理修复是否生效。

- **实际结果**：质量高——av_description 平均 678字，风格声明作首句 5/5、in/out_anchor 与 link_from/to、duration_hint 全部填充 5/5；角色锁定外貌**原样内嵌**（"周诚｜男性,40岁…眉头川字纹…青色胡茬…"）保障跨镜一致性；duration_hint 带依据（"12秒，含氛围铺陈/独白/发现新角色"）。
  - **瑕疵**：① av_description 平均 678字、最长 760字，**超出自设 ≤600字 上限**；② 内嵌用了 brief() 的"｜"分隔符（"周诚｜男性…"），在连贯散文里略生硬。

- **改善空间**：① 收紧到 600字（或放宽上限声明）；② 大项目分批/并行；③ 内嵌锁定描述时去掉"｜"改自然嵌入。

### 实验 24：shot-breakdown 重跑安全性 + 非确定性
- **目标**：验证二次拆镜是否安全（清旧写新、无孤儿/残留），观察重跑一致性。
- **方法**：对已有 16 镜的 proj_lab_taxi 再跑一次，核对 shots/details/char_links/orphan。
- **实际结果**：
  - **重跑安全 ✓**：重跑前 16 镜/16 详情/0 孤儿/15 角色关联 → 日志"已清除旧镜头 16 个" → 重跑后 13 镜/13 详情/**0 孤儿**/13 关联，details==shots（无残留）。旧镜头的角色关联随删 shot 级联清理，未累积孤儿（验证了代码注释里"依赖级联清理"的可靠性）。
  - **非确定性**：同一剧本三次拆镜得到 14（基线）/16/13 镜，数量与内容每次不同（temperature=0.6）。**"重跑微调"不可行**——重跑得到的是全新分镜板，非增量修改；用户若已在某版分镜上行内编辑过，重跑会全部丢失。
- **改善空间**：重跑前提示"将清空当前 N 个镜头及其编辑，确认？"；或提供"仅追加缺失场景镜头"的增量模式。
- **对新手**：清旧写新符合直觉，安全。
- **对老手**：老手若已精修分镜，误点重跑=前功尽弃，需强确认。
- **是否优秀产品**：数据安全合格；重跑语义（全量覆盖 vs 增量）需明确告知。
- **当场修正**：确认弹窗属前端，记待办。

## 待决策清单

按影响 × 成本排序：

1. **【高】全流程"长等待 + 无进度"体验**（实验 26/22）：文本三步串行 5-8 分钟、进度条恒 0%、逐个点按钮。建议：接真实进度条 + "一键跑通(extract→vdict→shot)" + 完成通知。属较大前端/编排改造。
2. **【高】视觉词典"锁定却不稳定"**（实验 14）：temp=0.7 重跑静默改写锁定描述，与"锁定=一致性根基"自相矛盾，且可能与已出图脱节。建议：锁定阶段调 temp 至 ~0.3 + 重跑前"将覆盖现有锁定"确认 / "仅补空缺"模式。需权衡描述丰富度。
3. **【中】commit-after-yield 时序缝隙的前端防御**（实验 10/18）：extract/PATCH 完成后立即读取会读到旧值，前端多处"刷新"可能显示陈旧/漏项。建议：写后读统一加短轮询重试封装。
4. **【中】分镜/词典阶段对画幅与风格无感知**（实验 1）：竖屏短剧(9:16)在拆镜/词典阶段无竖构图约束，直到出图才生效。建议：把画幅与 visual_style 下沉进 shot_breakdown / visual_dict 提示词。
5. **【中】重跑语义（全量覆盖 vs 增量）**（实验 24/14）：shot/extract/vdict 重跑都是全量重生成、非确定性，会丢失已有编辑。建议：破坏性重跑前强确认，长期提供增量模式。
6. **【中】专业分镜表字段缺失**（实验 21）：缺转场、音效/BGM 描述列；已有 vfx/atmosphere/分镜级画幅字段未暴露。建议：先暴露已有字段，再补转场/音效。
7. **【低】场景切分粒度**（实验 15）：同一物理空间的不同区域/时间点被过度切成多场景。extract 提示微调。
8. **【低】后端 shot-detail duration 无范围校验**（实验 18）：直连 API 可写越界值。建议后端加 1-60 校验。
9. **【低】三帧按运动量裁剪**（实验 23）：低动作镜三帧近似，可省 1/3 出图成本。
10. **【低】搜索召回浅**（实验 20）：不索引对白/实体描述。

## 总结论

### 当前产品最薄弱的 3 个环节
1. **分镜镜头语言贫乏（已当场修复）**：修复前运镜全 STATIC(100%)、机位全 EYE_LEVEL(93%)、对话零正反打——成片会像 PPT。这是最伤"成片质感"的环节。**本轮加硬约束后已降到 STATIC 25%、EYE 37-53%、出现成对过肩**，是本次最大增益。
2. **等待体验（未修，最伤留存）**：文本流水线串行 5-8 分钟 + 进度条恒 0% + 逐步点按钮，新手极易误判卡死而流失。这是"功能能跑但产品不好用"的核心矛盾。
3. **"锁定"的非确定性（未修，伤一致性根基）**：视觉词典/分镜重跑都会整份改写、非幂等。产品卖点是"从交接那刻锁死一致性"，但重跑机制自己在打破它。

### 本轮当场修正（均已 py_compile 通过）
- `bridge/extract_setup.py`：道具抽取加"三选一判据 + 固定构件排除清单"（出租车本 8→2 道具）；坏 pid 加"项目不存在"守卫（原抛裸 AttributeError）。
- `bridge/shot_breakdown.py`：加运镜/机位多样性 5 条硬约束（STATIC 100%→25%、过肩 0→4）；空镜"空镜："前缀改后处理（遵守率 0→5/5）；加空剧本守卫（省 GLM）。

### 建议路线图
- **第一优先（体验，两周内）**：进度条接真实状态 + "一键跑通"串联三步 + 完成通知；破坏性重跑强确认。—— 把"能跑"变成"敢用"。
- **第二优先（一致性）**：视觉词典锁定 temp 下调 + 重跑"仅补空缺"模式；写后读加轮询重试封装。
- **第三优先（专业度）**：画幅/风格下沉到分镜与词典；暴露 vfx/atmosphere/分镜级画幅；补转场与音效/BGM 列。
- **持续（成本）**：三帧按运动量裁剪、unit-gen 大项目分批、图像生成前置去重。

### 六问的横向结论
- **为何存在**：每一步都有清晰的存在理由，链路设计（剧本→设定→词典→分镜→帧）是专业且正确的。
- **改善空间**：主要在"提示词硬约束"（已验证极有效）、"确定性/幂等"、"等待与进度体验"。
- **对新手**：设定/分镜/词典的"一键专业产出"价值极高；但长等待+0%进度+重跑丢编辑是三大劝退。
- **对老手**：一致性锁定、可复用描述、行内编辑是真价值；缺增量重跑、专业分镜列、深度搜索。
- **是否优秀产品**：**内核（一致性契约 + 分步生产 + 锁定描述）是优秀的**；短板集中在"打磨层"（进度反馈、幂等、确认、字段暴露）。修完分镜镜头语言后，最大的下一步是把等待体验和重跑语义做扎实。
